# Azure Component Mapping

| Legacy component | Current-state role | Azure counterpart | Replacement type | Rationale | Open issue / caveat |
|---|---|---|---|---|---|
| SSIS orchestration | Batch ETL orchestration for file and SQL Server loads | Azure Data Factory pipelines | Functional | ADF provides scheduled orchestration, dependency control, and source connectivity for the same batch-style workflow. | Exact trigger timing, retry rules, and package-level dependency order still need validation. |
| SSIS staging tables | Temporary landing area before warehouse load | ADLS raw/staged zones plus Synapse staging models | Functional | The staging concept is preserved, but the physical form changes from SQL Server tables to lake zones and load-ready models. | Final staging schema and whether any SQL staging tables remain are still open. |
| SQL Server warehouse tables | Trusted business truth layer for admissions analytics | Azure Synapse Analytics warehouse tables / views | Functional | Synapse is the warehouse platform in the target design and should preserve the dimensional model and KPIs. | Deployment model and exact table layout are still open until detailed design is finalized. |
| Stored procedures | Encapsulated transformation logic for dimensions and facts | dbt models and SQL transformations in Synapse | Functional | dbt provides version-controlled, testable transformation logic that replaces opaque procedural ETL patterns. | Some procedural edge cases may still require SQL stored procedures or macros. |
| DimLocation SCD Type 2 logic | Historical preservation of province-to-city changes | dbt snapshot or incremental SCD logic | Functional | The business rule is preserved, but implementation shifts from legacy stored procedure logic to modern transformation patterns. | The exact snapshot strategy and change-detection rules are not yet confirmed. |
| FactAdmission accumulating metrics | Admission-level elapsed-time and completion tracking | Synapse fact model with dbt-derived metrics | Functional | The business meaning is preserved by rebuilding the admission fact and the elapsed-time calculations in the new model. | Column names, update sequence, and reconciliation logic must be validated carefully. |
| SSAS cube | Semantic model for trusted analytics and multidimensional analysis | Synapse views / marts plus Power BI semantic consumption | Partial | The reporting interface can be recreated, but there is no perfect one-to-one Azure equivalent for the legacy cube behavior. | Cube measures, hierarchies, and calculated members remain open; semantic parity needs testing. |
| Excel pivots | Ad hoc analysis by business users connected to SSAS | Excel connected to the new semantic layer | Functional | Excel can continue to consume the new semantic outputs in a familiar way. | Workbook inventory, pivot dependencies, and user-specific formatting need validation. |
| Power BI reports | Dashboard and reporting consumption layer | Power BI reports on Synapse views or marts | Functional | Power BI remains the target consumption tool and can be pointed at the new warehouse layer. | Direct vs semantic-model connection pattern is intentionally not fixed yet. |
| Data profiling | Staging quality review before load | dbt tests and pipeline validation checks | Functional | Data validation is preserved, but moved into explicit tests and pipeline checks. | Exact legacy profiling outputs are not fully documented and need review if parity is required. |
| Semantic behavior / drill-through | Trusted slicing, drill paths, and report navigation | Power BI model behavior plus curated views/marts | Partial | The user experience can be approximated, but drill-through and cube-specific semantics may not match exactly. | The report confirms drill-through use, but not the full semantic design, so parity testing is required. |
